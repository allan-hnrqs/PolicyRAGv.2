# Persistent Optimization Loop Protocol

This repo now has a concrete operating layer for the persistent optimization
loop. The goal is to keep cycling until the combined tiered product reaches the
quality target or a documented hard blocker is found.

## Control And Persistent Surfaces

- control profile:
  - `baseline_vector_rerank_shortlist`
- control surface manifest:
  - `datasets/eval/manifests/optimization_control_surface_v1.json`
- persistent failure surface manifest:
  - `datasets/eval/manifests/persistent_failure_surface_v1.json`
- conversational review surface:
  - `datasets/eval/manifests/conversational_review_surface_v1.json`

The control surface is frozen. The failure surface is allowed to evolve, but
only when the control profile and failure pattern change materially.

There is now also a broader diagnostic/acceptance surface:
- `datasets/eval/manifests/acceptance49_blueprint.json`
- `datasets/eval/manifests/acceptance49_additions_batch1.json`
- generated by `scripts/build_acceptance49_working.py`

Use this wider surface to expose additional failure families and to test
whether a candidate win generalizes beyond the frozen control bank. Do not
silently substitute it for `parity19`.

## Loop Types

- large cycle:
  - one architectural hypothesis
  - canonical dev/holdout
  - retrieval-only
  - product benchmark
  - multiturn if relevant
- small cycle:
  - one truth-maintenance task
  - failure-surface benchmarking
  - diagnostics, benchmark governance, or cleanup

Never run two large cycles back-to-back without a small cycle in between.

## Cycle Artifact

Use:

```powershell
python scripts\run_optimization_cycle.py --cycle-id loop01_small --hypothesis-id packing_audit --profile baseline_vector_rerank_shortlist --cycle-kind small
```

The cycle runner writes:
- JSON artifact under `datasets/runs/optimization_cycles/`
- Markdown summary under `datasets/runs/optimization_cycles/`

Each cycle artifact records:
- cycle id and hypothesis id
- classification
- materialized failure surfaces
- benchmark artifacts
- external comparator references

## Classification Rule

Every large cycle must be classified:
- `promising`
- `mixed`
- `rejected`

Do not leave serious large-cycle results undocumented.

## Subagent Use

When subagents are used, keep the roles bounded:
- retrieval auditor
- benchmark auditor
- single-subsystem implementer

Do not let subagents overlap write scopes. Do not delegate the architecture
decision itself.

## Anti-Reward-Hacking Rule

- keep `parity19` frozen as control
- do not tune on holdout
- do not promote from the failure surface alone
- exactness regressions veto promotion
- benchmark edits must be documented with rationale and examples
